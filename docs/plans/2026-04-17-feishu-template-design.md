# 飞书推送模板可配置设计文档

**日期：** 2026-04-17

## 1. 目标

把飞书日报推送文案从代码中抽出来，改成后台可配置模板，方便你后续：

- 调整推送标题风格
- 控制正文结构
- 让推送内容跟日报编辑后的标题与导语保持一致

本轮只做日报推送模板，不做：

- 飞书机器人多套模板切换
- 图文卡片高级布局编辑器
- Markdown / 富文本解析

## 2. 方案对比

### 方案 A：继续写死在代码里

优点：

- 改动最少

缺点：

- 每次改文案都要改代码
- 不适合持续运营

### 方案 B：系统设置里保存标题模板和正文模板

优点：

- 实现简单
- 复用现有 `system_settings`
- 管理后台可直接改

缺点：

- 模板能力有限
- 只支持预设变量替换

### 方案 C：存 JSON 结构化飞书卡片模板

优点：

- 灵活度最高

缺点：

- 对当前项目过重
- 容易把设置页变复杂

## 3. 采用方案

本轮采用 **方案 B**。

## 4. 模板设计

新增两个设置项：

- `feishu.report_title_template`
- `feishu.report_body_template`

默认值：

- 标题：`{{report_title}}`
- 正文：

```text
日期：{{report_date}}
导语：{{report_intro}}
共整理 {{article_count}} 篇文章，覆盖 {{source_count}} 个来源。
今日看点：
{{highlights_bullets}}
完整阅读：{{report_url}}
```

## 5. 支持变量

本轮支持这些占位符：

- `{{report_title}}`
- `{{report_date}}`
- `{{report_intro}}`
- `{{article_count}}`
- `{{source_count}}`
- `{{report_url}}`
- `{{highlights_bullets}}`

其中：

- `highlights_bullets` 是多行文本，形如：
  - `• 标题 A`
  - `• 标题 B`
  - `• 标题 C`

## 6. 后端设计

### 6.1 模板渲染

新增一个轻量模板渲染函数，对固定变量做安全替换，不引入 Jinja。

原因：

- 模板场景简单
- 避免模板执行能力过强
- 更容易控制变量范围

### 6.2 推送构造

推送日报时：

1. 读取日报记录
2. 收集 highlights
3. 构造模板上下文
4. 渲染标题模板
5. 渲染正文模板
6. 以 Feishu `post` 消息格式发送

正文按换行拆成多段文本块。

### 6.3 测试消息

测试消息接口保持不变，仍然允许手动输入标题和正文。

本轮不额外做“按模板发送测试”接口，避免 API 膨胀。

## 7. 前端设计

在系统设置页中增加一块“飞书日报模板”：

- 标题模板输入框
- 正文模板文本域
- 变量说明
- 示例预览

示例预览用前端本地替换样例数据即可，不需要额外后端接口。

## 8. 风险与处理

### 8.1 模板变量拼错

不会报错中断，未识别变量会原样保留，便于发现问题。

### 8.2 正文空行过多

渲染发送时过滤纯空白行，避免飞书内容太松散。

### 8.3 导语为空

变量替换为空字符串，不额外报错。

## 9. 验证

需要覆盖：

- 默认模板设置初始化
- 模板渲染结果正确
- 日报推送使用数据库模板而不是硬编码
- 设置页可保存模板
- 前端构建通过
