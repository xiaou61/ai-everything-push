# 日报编辑后再发布设计文档

**日期：** 2026-04-17

## 1. 目标

在现有“自动生成日报 -> 公开页面 -> 飞书推送”链路上，补一层人工审稿能力，让后台可以：

- 查看某天日报的完整条目明细
- 修改日报标题和导语
- 调整日报内文章顺序
- 修改条目分组名
- 从日报中移除不想展示的文章
- 保存后重新发布公开 HTML 页面

本轮不做：

- 从文章池手动往日报里新增文章
- 可视化拖拽排序
- 富文本编辑
- 审批流

## 2. 方案对比

### 方案 A：最小编辑器

基于现有 `daily_reports` 和 `daily_report_items` 直接做后台编辑器，只编辑已入选日报的条目集合和排序。

优点：

- 改动范围小
- 数据模型不用扩展
- 能最快打通“编辑后再发布”

缺点：

- 不能从文章池补新条目
- 编辑能力偏轻量

### 方案 B：日报编排器

后台支持从文章池检索文章、加入日报、移除条目、批量改分组，接近一个完整 CMS。

优点：

- 灵活度更高

缺点：

- 本轮实现成本明显更高
- 需要额外设计筛选、冲突和候选池逻辑

### 方案 C：直接编辑生成 HTML

不改数据库，只允许编辑最终 HTML 文件。

优点：

- 后端 ORM 改动少

缺点：

- 数据与页面状态分离
- 后续重新推送和重新渲染很难维护

## 3. 采用方案

本轮采用 **方案 A：最小编辑器**。

原因：

- 和当前系统结构最一致
- 可以直接复用 `daily_reports` / `daily_report_items`
- 足够支撑“先自动汇总，再人工润色”的真实工作流

## 4. 后端设计

新增 3 类能力：

### 4.1 日报详情接口

提供单份日报详情，返回：

- 日报基础信息
- 条目列表
- 每条文章的来源、标题、摘要、分类、原文地址

### 4.2 日报编辑接口

允许保存：

- `title`
- `intro`
- `items[]`

其中 `items[]` 每项包含：

- `id`
- `display_order`
- `section_name`

如果某条 `daily_report_item` 没有出现在提交列表中，则视为从日报中移除。

### 4.3 重新发布接口

重新读取数据库中的日报和条目，重新渲染 HTML 文件，并更新：

- `html_path`
- `html_url`
- `article_count`
- `source_count`
- `status`

飞书推送仍然沿用现有“按日期推送”的接口，不和发布动作自动绑定。

## 5. 前端设计

在“日报发布台”页面增加一个编辑工作区：

- 左侧保留日报历史卡片
- 点击“编辑日报”后，右侧加载日报详情
- 顶部编辑标题、导语
- 下方按条目列表显示文章
- 支持：
  - 上移
  - 下移
  - 修改分组
  - 移除
- 底部提供：
  - 保存改动
  - 重新发布
  - 推送飞书

视觉上保持当前 editorial 风格，不新建独立路由，减少跳转成本。

## 6. 数据流

1. 用户在日报页选择一份历史日报
2. 前端请求日报详情
3. 用户修改标题、导语、顺序、分组或移除条目
4. 点击保存，前端提交结构化 payload
5. 后端更新 `daily_reports` 与 `daily_report_items`
6. 用户点击重新发布
7. 后端重新渲染 `docs/reports/YYYY-MM-DD.html`
8. 用户再决定是否推送飞书

## 7. 风险与处理

### 7.1 顺序错乱

保存前由前端重新生成连续的 `display_order`，后端再统一归一化一次。

### 7.2 移除条目后统计不准

每次保存或重新发布时，统一按当前条目重算 `article_count` 和 `source_count`。

### 7.3 编辑后公开页没有更新

提供独立“重新发布”动作，显式触发 HTML 重渲染，不依赖保存自动刷新文件。

## 8. 验证

需要覆盖：

- 日报详情接口
- 保存标题/导语/分组/顺序
- 移除条目后计数更新
- 重新发布后公开页出现编辑后的标题和导语
- 现有日报生成流程不回归
