# 来源健康监控与失败重试设计

## 背景

当前项目已经具备内容源抓取、文章处理、日报生成、飞书推送和后台管理能力，但来源抓取状态仍停留在“最后一次成功/失败”的基础层面。随着来源数量增加，仅靠 `last_crawl_status` 和 `consecutive_failures` 很难快速判断：

- 哪些来源只是短时抖动，应该自动重试
- 哪些来源已经进入冷却期，当前不该继续打
- 哪些来源已经持续失败，需要在后台明确告警
- 哪些来源长期未抓取，需要和失败来源区分开

这一轮目标不是做复杂的监控系统，而是补齐一个可扩展的最小闭环。

## 目标

实现以下闭环能力：

1. 抓取失败后支持有限次自动重试
2. 将重试策略与健康判定从抓取主流程中拆出
3. 来源 API 返回健康状态、冷却时间和最近成功/失败信息
4. Dashboard 展示异常来源摘要
5. Sources 页面可直接筛出健康、异常、冷却中的来源
6. README 改成中文开源展示型说明

## 不做的内容

本轮明确不做：

- 登录鉴权
- 事件流表 / 监控历史表
- 告警中心
- 单来源手动重试按钮
- 可配置的复杂退避策略
- 多实例分布式任务协调

## 方案选择

### 方案 A：只在现有字段上硬编码判断

优点：

- 改动最少

缺点：

- `crawl_service.py` 会继续膨胀
- 前后端都需要复制判断逻辑
- 后续很难继续加冷却、告警、单来源重试

### 方案 B：新增少量字段 + 独立健康策略服务

优点：

- 改动成本低
- 逻辑可复用
- 方便后续扩展更多运行策略

缺点：

- 需要补 schema 升级与序列化逻辑

### 方案 C：新增抓取事件表，完整记录每次尝试

优点：

- 可做更完整监控与统计

缺点：

- 本轮明显过重
- 会拉长实现、测试和前端适配成本

### 结论

采用方案 B。

## 数据模型设计

在 `sources` 表上追加以下字段：

- `last_success_at`
- `last_failure_at`
- `last_retry_attempts`

继续复用已有字段：

- `last_crawled_at`
- `last_crawl_status`
- `consecutive_failures`
- `last_crawl_error`
- `last_crawl_processed_count`

不新增事件表。健康状态与下次重试时间采用派生计算，不落库。

## 健康状态设计

来源健康状态统一收敛为五类：

- `idle`：尚未抓取
- `healthy`：最近成功且当前无失败积压
- `warning`：最近抓取有部分成功，或失败次数较少但还未进入冷却
- `cooling`：失败后仍处于重试冷却窗口
- `failed`：连续失败已超过策略阈值，或不满足自动重试条件

同时返回中文标签，前端不再自己拼装状态文案。

## 重试策略设计

新增独立服务 `app/services/source_health_service.py`，负责以下职责：

- 判断错误是否适合自动重试
- 根据重试次数生成下一次可重试时间
- 基于来源状态计算健康等级
- 统一回写成功与失败字段
- 对外提供序列化辅助函数，保证来源列表和详情返回一致

基础策略：

- 最多自动重试 3 次
- 使用短退避窗口，默认 `2 / 5 / 15` 分钟
- 可重试错误：
  - 超时
  - 连接失败
  - HTTP `429`
  - HTTP `5xx`
- 不可重试错误：
  - HTTP `404`
  - 明显配置或规则错误

## 抓取流程调整

`crawl_service.py` 只保留抓取编排职责：

1. 加载来源列表
2. 跳过未到重试窗口的来源
3. 调用抓取与入库逻辑
4. 成功时调用 `apply_source_crawl_success`
5. 失败时调用 `apply_source_crawl_failure`
6. 汇总作业结果与错误信息

这样可以把“怎么判定失败”和“失败后如何回写”与抓取主流程解耦。

## API 返回设计

### 来源接口

`SourceRead` 新增：

- `last_success_at`
- `last_failure_at`
- `last_retry_attempts`
- `next_retry_at`
- `can_retry_now`
- `health_level`
- `health_label`

### Dashboard 接口

在 `/admin/api/dashboard` 中新增：

- `source_health_summary`
- `source_alerts`

用于展示：

- 异常来源总数
- 冷却来源总数
- 最近失败来源列表

## 前端设计

### Sources 页面

新增：

- 健康筛选：全部 / 健康 / 异常 / 冷却中
- 健康状态中文徽标
- 最近成功时间
- 最近失败时间
- 下次可重试时间
- 上次重试次数

视觉上保持当前控制台风格，只增强信息密度和状态提醒。

### Dashboard 页面

新增异常来源提醒卡片，显示：

- 异常来源数量
- 冷却来源数量
- 最近失败来源

## 测试策略

先补测试，再做实现。覆盖重点：

- 可重试错误会生成重试计划
- 不可重试错误不会重试
- 成功回写会清空失败计数并记录成功时间
- 失败回写会记录失败时间和重试次数
- 来源接口返回新增健康字段
- Dashboard 返回异常来源摘要

## 可扩展性说明

这一轮设计会为后续扩展预留清晰位置：

- 更复杂的退避策略
- 单来源手动重试
- 失败来源告警推送
- 健康评分
- 来源 SLA 统计

核心原则是：抓取主流程编排、健康策略计算、前端展示状态三层分离。
