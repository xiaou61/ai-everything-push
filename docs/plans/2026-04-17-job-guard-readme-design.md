# 作业互斥与 README 补全设计文档

**日期：** 2026-04-17

## 1. 目标

这一轮聚焦两个方向：

- 给抓取、处理、生成日报、推送飞书四类作业补统一的运行互斥保护
- 把 README 补成一份能直接指导本地启动、配置、使用和排障的完整说明

额外要求：

- 手动触发和定时触发共用同一套互斥逻辑
- 冲突时接口返回明确错误，而不是静默重复执行
- 最后进行本地接口烟测

## 2. 方案对比

### 方案 A：只依赖 APScheduler `max_instances`

优点：

- 改动最小

缺点：

- 只能约束调度器内部
- 不能覆盖手动接口调用
- 不能给前端返回明确的“任务已在运行中”

### 方案 B：只用数据库检查 `JobRun` 里是否存在 running 记录

优点：

- 能跨手动与调度统一判断
- 有运行记录可追踪

缺点：

- 如果异常退出，容易留下脏的 running 记录
- 单靠查询判断，边界上仍可能重复进入

### 方案 C：进程内锁 + 数据库运行记录双保险

优点：

- 本地单实例运行时足够稳
- 手动和调度都能统一复用
- 冲突返回明确，且保留数据库审计记录
- 可顺手补“陈旧 running 任务自动超时”

缺点：

- 主要面向当前单机部署，不是分布式锁

## 3. 采用方案

本轮采用 **方案 C**。

原因：

- 当前项目是本地优先、单机部署为主
- 用最小复杂度解决最现实的问题
- 后续如果真要多实例部署，再升级为数据库锁或 Redis 锁即可

## 4. 设计细节

### 4.1 统一作业运行守卫

新增一层作业守卫服务，负责：

- 按 `job_name` 申请非阻塞运行锁
- 创建 `JobRun(status="running")`
- 检测同名运行中任务
- 自动把过旧的 running 记录标记为 `timeout`
- 作业结束时统一回写状态并释放锁

### 4.2 冲突处理

当同名任务已经在运行：

- 手动 API 返回 `409 Conflict`
- 调度器只记录日志并跳过本次执行

### 4.3 幂等策略

本轮不做复杂去重算法，先补最小幂等：

- 同一时刻同一类任务不允许重复运行
- 调度器触发的飞书推送，如果当天日报已标记 `feishu_pushed=true`，则直接跳过

说明：

- 手动推送仍保留人工重发能力
- 日报生成本身已按日期覆写，天然具备基础幂等

### 4.4 README 补全范围

README 需要覆盖：

- `pytest` 全量回归

## 5. 风险与处理

### 5.1 锁释放不及时

处理：

- 统一使用 `try/finally`
- 补陈旧 running 记录自动超时

### 5.2 手动和调度返回语义不同

处理：

- API 层返回 409
- 调度器记录 skip 日志，不抛出未处理异常

### 5.3 README 过长但信息仍不全

处理：

- 以“新人从零跑通”为标准写
- 用章节化结构，不只写启动命令

## 6. 验证

需要覆盖：

- 同名任务冲突时返回 409
- 同名任务正常结束后可再次运行
- 调度器已推送日报时自动跳过二次推送
- README 覆盖环境变量、运行方式、接口与排障说明
- 本地服务启动后主要接口返回正常
